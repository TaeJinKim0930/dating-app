### 1-3. 이동수단(모드) 설계 — 이번 버전의 코드 변경 핵심
```js
// PLACES 항목에 leg(이 장소까지 오는 구간) 정보 추가
{ name:'브런치빈', ..., leg:{ mode:'car', km:7, min:14, note:'시내 주행' } }
// mode: 'car' | 'transit' | 'walk'  → 아이콘(🚗🚌🚶), 선 스타일, 딥링크가 모두 이걸 따름
```
- **네이버 길찾기 딥링크 모드 분기**: `map.naver.com/p/directions/<from>/<to>/-/car` (기존 `/walk`를 leg.mode로 치환)
- **지도 오토핏**: v1은 부평 500m 스케일 고정(K=0.72). 을왕리 코스는 폭 ~30km → `buildMap()`이 장소 bounding box를 계산해 viewBox에 **자동 맞춤**하도록 변경 (이후 어떤 코스든 재사용 가능)
- 거리·시간은 실측치(leg에 명시)를 우선 사용, 없으면 직선거리로 추정 (차량은 직선×1.35, 45km/h 기본)

### 1-4. 장난 인터랙션 팩 (리서치 20개 중 추천 셀렉션)
> 원칙: **한 페이지에 3~5개만.** 다 넣으면 '음..' 버튼의 임팩트가 죽는다.

| 추천 | 이름 | 내용 | 난이도 |
|---|---|---|---|
| ⭐ | **기어 시프트 RSVP** | P·R·D 기어 레버 드래그로 답변. D=출발!+하트폭죽, R(후진=거절)은 "삐빅— 이 차량은 후진을 지원하지 않습니다" 하고 D로 미끄러짐. '음..' 버튼의 정신적 후계자 | 중 |
| ⭐ | **미니카 경로 주행** | 지도 경로 위를 미니 자동차가 스크롤 따라 주행, 정거장 도착 시 핀 통통 | 중 |
| ⭐ | **내비 안내 멘트** | 구간 진입 시 내비 말풍선: "경로에 바다가 포함되어 있습니다. 설렘 주의" / 마지막 "목적지는 당신의 마음입니다" | 하 |
| ⭐ | **노을로 물드는 배경** | 스크롤 위치 따라 아침→오후 바다→을왕리 구간에서 노을 그라디언트+해가 수평선으로 | 하 |
| 선택 | 자동차 키 시동 오프닝 | 봉투 대신 키 꽂고 드래그로 돌려 시동 → 헤드라이트 켜지며 초대장 등장 (봉투 유지 여부 선택) | 중 |
| 선택 | 도망 버튼 2.0 항복 엔딩 | 7번 쫓으면 "알겠어요, 졌어요..." 하며 '네' 버튼으로 변신 | 하 |
| 백로그 | 김서린 차창 닦기, 와이퍼 물리치기, D-day 계기판, 경적 이스터에그, 안전벨트 게이트, 데이트 견적 영수증, 탑승권 뜯기, 갈매기 새우깡, 하이패스 전환, 인생네컷 예약석, 스탬프 랠리, 설렘 주유 게이지 | 다음 버전들에서 로테이션 | — |

## 2. 제품 아키텍처 — "코드 수정 없이 웹에서 만드는" 구조

### 2-1. 3개 화면(레이어)로 분리
```
┌─ Viewer (초대장) ──── 지금 있는 것. 코스 데이터를 받아 렌더
├─ Builder (에디터) ─── 신규. 지도에서 장소 검색/추가 → 링크 생성
└─ Wishlist (위시리스트) ─ 신규. 가고 싶은 곳 저장 → Builder 재료
```

### 2-2. 핵심 트릭: **URL이 곧 데이터베이스** (DB 없이 GitHub Pages 유지)
```
초대장 링크 = index.html#c=<코스JSON을 lz-string으로 압축한 문자열>
```
- Builder에서 만든 코스 전체(장소·시간·모드·문구·테마)를 **URL 해시에 압축 인코딩** → 서버·DB 0원, 링크 자체가 완결된 초대장
- 받는 사람 열람 시 지도 API 호출 없음(자체 SVG 렌더) → **바이럴로 조회수 폭증해도 원가 0** (시장조사의 'API 원가 폭탄' 함정 회피)
- 지금의 `?to=&from=` 파라미터는 그대로 호환 유지

### 2-3. 장소 검색 (Builder 안)
| 단계 | 방법 | 키 필요 |
|---|---|---|
| MVP | 지도 클릭으로 핀 찍기 + 이름/네이버링크 수동 입력 | ❌ |
| 확장 | **카카오 로컬 REST API** (키워드 장소검색, 무료 쿼터 일 10만+) | 무료 키 |
- 배경지도: MVP는 자체 귀여운 SVG(현행 스타일) / 확장 시 Leaflet+OSM 타일(무료)을 Builder에만 사용, Viewer는 계속 자체 렌더

### 2-4. AI 코스 추천 (예: "홍대 데이트 짜줘", "내 위시리스트로 코스 만들어줘")
정적 페이지에 LLM API 키를 넣으면 노출됨 → 방법 3가지:
| 방법 | 설명 | 판정 |
|---|---|---|
| **Cloudflare Workers 무료 프록시** | 키를 서버리스에 숨기고 일일 호출 제한. 무료 티어 10만req/일 | ⭐ 권장 (Phase 3) |
| BYO-key | 사용자가 자기 API 키 입력 | 데모용으론 가능, 대중용 아님 |
| 미리 만든 코스 갤러리 | AI 없이 UGC 템플릿으로 대체 | Phase 2에서 먼저 |
- AI 입력 = 위시리스트(좌표 포함) + 지역/날짜/이동수단 → 출력 = 코스 JSON(그대로 Viewer 렌더) → **초대장 자동 생성까지 묶는 것**이 카카오 'AI 메이트 로컬'과의 차별점

### 2-5. 위시리스트
- Phase 2: **localStorage** (기기 내 저장, DB 불필요) — 담기/보기/Builder로 보내기
- Phase 4: Supabase(무료 BaaS)로 계정·기기 동기화·커플 공유 (이별 시 데이터 분리 정책도 이때 설계)

## 4. 시장 인사이트 요약 (조사 기반 — 방향성 근거)

### 성공 공식
1. **받는 사람이 다음 만드는 사람이 되는 루프**: 초대장 열람 → 감탄 → RSVP → "나도 만들기" CTA → 가입 없이 제작. (모바일 청첩장이 검증한 경로. 각 단계 전환율이 핵심 지표)
2. **저빈도(코스 짜기) 사이를 메꾸는 고빈도 훅**: 썸원은 '하루 1질문'만으로 DAU 130만. → 디데이·위시리스트·"이번 주말 뭐할까"가 리텐션 담당
3. **옮기기 아까운 자산**: 보낸 초대장 + 다녀온 코스 + 사진 = "우리의 데이트 아카이브" (디데이 앱 수십 개가 못 만든 차별점)
4. **UGC 갤러리로 지방 커버리지**: 데이트팝도 직접 큐레이션으론 5대 광역시 한계 → 유저 코스를 익명 템플릿화

### 피해야 할 함정
1. **트래픽≠수익** (비트윈: MAU 120만에도 3번 매각) → 수익화는 처음부터 설계
2. **지도 거인과 기능 경쟁 금지** (네이버·카카오가 저장·공유·AI추천 다 함) → 우리 것은 "관계 맥락 + 감성 연출 + 초대 의례 + 아카이브"
3. **지도 API 원가 폭탄** → 열람 페이지는 자체 렌더(§2-2로 해결)
4. **'데이팅 앱' 오인 포지셔닝** → 스토어 카테고리는 라이프스타일/커플 플래너, 문구에 '데이트 코스·초대장' 명시
5. **이별 이탈 + 연애인구 감소** → 브랜드를 '데이트'에 못박지 말 것: 친구 생일 코스·부모님 효도 코스·우정 여행에도 통하는 "누군가를 위해 하루를 설계해 초대하는 서비스" (Cobble이 그룹 플래너로 피벗한 이유)

### 수익화 우선순위
1. **프리미엄 템플릿·연출 단건 결제** (봉투 스킨·계절 한정·음악, 2천~1만원) — DAU 없이도 성립, 청첩장/썸원골드가 지불의사 검증
2. 커플 멤버십 (월 3~5천원: 무제한+AI+아카이브)
3. AI 추천 크레딧 (LLM 원가와 자연 연동)
4. 제휴(캐치테이블 예약·팝업 티켓·기념일 꽃/케이크) — 트래픽 확보 후. **직접 커머스 운영은 금지**(비트윈·HowAboutWe 실패 지점)
5. 광고 — DAU 수십만 전까지 계획에서 제외
